# 证据库拆条任务规范（给执行拆条的 AI）

> 你的任务：把 `backend/evidence/raw/` 里的**权威原始文档**拆成结构化「结论条目」，输出到 `backend/evidence/entries/`。
> 主工程师只负责定规范和验收，具体拆条由你执行。请严格按本规范，产出格式必须统一。

## 0. 铁律（违反即作废，最重要）

1. **只拆 `raw/` 目录里的文件。** `raw/` 里放的都是人工下载的真实权威原件（指南、Cochrane 综述、WHO/机构官方文件）。
2. **绝对不要拆 `deepresearch/` 目录的任何内容。** 那里是 AI 生成的检索报告，只用于找线索，**AI 生成内容严禁进入证据库**——这是本产品可信度的红线。
3. **不得编造原文没有的内容。** 每条结论必须能在原文里找到依据。原文没说的数字、机构、结论，一个字都不许加。
4. 拆条只做"提炼与结构化"，不做"补充与发挥"。你是搬运工，不是作者。

## 0.5 环境准备（每台要跑拆条的电脑，首次做一次）

拆条脚本要调用 DeepSeek，所以运行它的电脑需要配 API 密钥：

1. 装依赖（首次）：`pip install openai pymupdf rapidocr-onnxruntime python-dotenv`
2. 申请密钥：去 https://platform.deepseek.com 注册 → 充值（拆条很便宜，几十元够拆很多本）→ 创建 API Key。
3. **填密钥（最简单）**：用记事本打开 `ingest_evidence.py`，在文件顶部找到 `API_KEY = ""`，把密钥填进引号里，例如 `API_KEY = "sk-1234..."`，保存。就这一步。

> 多人分工时：谁的电脑跑拆条，谁就在自己那份脚本顶部填自己的密钥。只负责收集 PDF、不跑脚本的人不需要密钥。
> 填了密钥的脚本别提交到 git（各自本地用即可），避免密钥外泄。

## 1. 工具与方法

仓库已有拆条脚本 `backend/ingest_evidence.py`，它能处理文字版 PDF，也能对扫描版 PDF 自动 OCR。

### 推荐：批量模式（一条命令跑完清单）

不用为每份 PDF 手搓命令。所有待拆文档登记在 **`backend/evidence/registry.csv`**（列：`filename,org,doc,year,url,topic,pages`），一条命令全跑：

```bash
cd D:/PointMap/backend
python ingest_evidence.py --manifest evidence/registry.csv
```

脚本会逐行读 CSV，对 `evidence/raw/` 里对应的 PDF 拆条，输出到 `evidence/entries/`。
**你（执行拆条的 AI）要做的**：收到新 PDF 后，在 `registry.csv` 追加一行（`pages` 列填正文核心页范围，先翻 PDF 目录确定，跳过封面/前言/附录/食谱；留空则整本），然后跑上面这条命令即可。

### 单份模式（临时测一份用）

```bash
python ingest_evidence.py "evidence/raw/文件名.pdf" \
    --org "发布机构" --doc "文档全名" --year 2022 \
    --url "官方链接" --topic "领域" [--pages 起-止]
```

若某文档不适合该脚本（如英文文献、网页导出的 HTML/Markdown），你可自行用等效方法处理，但**输出的 JSON 结构必须和下面第 2 节完全一致**。

⚠️ **两个关键提醒（否则又慢又费 token）**：
1. **是"运行脚本"，不是让你逐页读 PDF。** 脚本用 Python 抽文字/OCR + 调 API 拆条，你只需执行命令。不要自己把整本 PDF 读进对话里手动拆——那样又慢、又贵、又不准。
2. **拆条一律用快模型 `deepseek-chat`（脚本已默认），不要用推理模型 `deepseek-v4-pro`。** 拆条是机械的阅读理解，推理模型会慢 5~10 倍、token 翻好几倍且质量无提升。（推理模型只留给正式的视频分析。）

## 2. 输出格式（所有条目必须长这样）

输出到 `backend/evidence/entries/{文档短名}.json`：

```json
{
  "doc": "中国居民膳食指南（2022）",
  "org": "中国营养学会",
  "year": "2022",
  "url": "http://dg.cnsoc.org/...",
  "count": 55,
  "entries": [
    {
      "id": "E-CJDG-001",
      "claim": "健康孕妇每天可摄入1个全蛋，且不应丢弃蛋黄。",
      "section": "准则四 适量吃鱼、禽、蛋、瘦肉",
      "strength": "指南推荐",
      "topics": ["孕期","鸡蛋","蛋黄","胆固醇"],
      "source_doc": "中国居民膳食指南（2022）",
      "org": "中国营养学会",
      "year": "2022",
      "url": "http://dg.cnsoc.org/...",
      "page": "78-79"
    }
  ]
}
```

字段规则：
- **id**：`E-<文档短代码>-<三位序号>`。文档短代码用拼音首字母，如 中国居民膳食指南=CJDG、孕期妇女膳食指南=YQFN。同一文档内序号连续。
- **claim**：把原文结论改写成**一句面向核验的明确陈述**（能直接拿去判断"某视频说法对不对"）。不要照抄整段。
- **section**：结论所在的准则/章节名（原文能定位就填，不确定留空）。
- **strength**：从 `["指南推荐","指南建议","一般共识","背景说明"]` 选最贴切的。综述类文献可用"高级别证据"。
- **topics**：2~5 个检索关键词。
- **page**：原文页码范围（PDF 用得上；非 PDF 可留空）。

## 3. 只拆有"结论"的内容

✅ 拆：明确的推荐、建议、量化标准、"应该/不应该/推荐/不推荐"式的结论。
❌ 跳过：目录、前言、致谢、纯背景介绍、无结论的叙述、食谱、参考文献列表。

## 4. 自查（交付前必做）

- 随机抽 10~20% 条目，回原文核对：claim 是否真实、有没有夸大或编造。
- 检查 URL 字段是否为真实官方链接。
- 确认没有拆到 `deepresearch/` 的内容。
- 交付时附一句：本批处理了哪个文档、共多少条、抽查了哪几条、有无存疑。

## 5. 交付后

产出留在 `backend/evidence/entries/`，通知主工程师验收。主工程师会做二次抽检 + 去重 + 建检索索引入库。**你不需要建索引或改后端。**
