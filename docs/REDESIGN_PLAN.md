# FitProof 大区赛系统改造方案（给程序员 AI 的总纲）

> 本文件是改造的总设计。程序员 AI 动手前必须读完本文件 + `CONTEXT_FOR_NEW_AI.md`。
> 架构由主设计师定，你负责按任务实现。每个任务做完通知负责人，由主设计师审代码 + 验效果。

## 一、改造总目标

把现在这个「双视频输入、只能本地跑、判定靠前端正则猜」的系统，改造成
「**单视频为主、云端可跑、判定由大模型基于真实权威证据给出**」的健康信息核验工具。
面向人群从"健身人群"扩为"所有关注健康、爱刨根问底的人"（孕妇/宝妈/慢病/中老年…）。

**核心资产保留**（不要推翻）：6 张核验卡的 UI 与交互、后端视频提取管线、ResultPage 的命令式拖拽。
**改造集中在**：输入侧、分析层、证据层。

## 二、必须遵守的项目铁律（违反即返工）

1. **数据驱动，绝不写死话题文案**——组件对任意 analysis JSON 都要正确渲染。
2. **不编造权威来源**——只引用真实机构/指南，绝不编论文/DOI/数字。
3. **DeepSeek 推理模型（deepseek-v4-pro）所有调用 `max_tokens >= 8192`**，否则 content 返回空。拆条等机械任务才用快模型（deepseek-v4-flash）。
4. **判定/依据强度/风险必须由模型输出，不再用前端正则启发式**（现 ResultPage 里那套 `conflictRisk`/`riskPointForMisleading` 等关键词正则要废弃）。
5. ResultPage 两个易复现 bug 的实现别动：命令式拖拽（不 setState）、卡片用 `renderCard()` 函数而非组件（防输入框失焦）。
6. 改前端跑 `npx tsc --noEmit` 自查；改后端至少 `python -c "import main"` 能过再回报。
7. 不在 dev server 运行时跑 `next build`（会写坏 .next 缓存）。

## 三、两种核验模式（核心产品逻辑）

- **单视频（主）**：视频 → 转写+画面OCR → 快模型拆出「可核验主张清单」（3~5 条，每条标信号：疑似夸大/有条件/较公认/有争议）→ 用户点选一条 → 检索证据库 → 推理模型深度核验，对面站的是**主流权威证据**。
- **双视频（次，保留）**：跨视频提取共识/分歧，证据当裁判，对面是**另一个视频**。分歧是视频本就存在的，AI 不制造对立。
- 两种模式共用同一套核验卡与引用体系。**绝不做"AI 编个反方来辩论"的假辩论。**

## 四、证据库与 RAG（已有 606 条真实数据）

- 证据条目已由 `ingest_evidence.py` 拆好，存在 `backend/evidence/entries/*.json`。每条含：
  `id, claim, section, strength, topics, source_doc, org, year, url, page`。
- 核验时：主张 → 检索证据库 → 命中条目注入 prompt → **模型只许引用注入的条目**（引用其 id/来源），每条判定可点击溯源到指南原文。
- **三档透明降级**：①命中权威条目 → 引用[n]；②未命中 → 照给判断但明确标注"AI 常识判断，未命中已收录依据"并降低依据强度；③非健康话题 → 坦然拒绝。
- 检索技术选型：**本地中文向量模型 `bge-small-zh`**（纯本地、免费、毫秒级）+ 轻量存储，不引入重型向量数据库。

### 任务 1：证据检索层（独立可测）

**目标**：做一个能对证据库做语义检索的模块，供后续核验调用。**先独立成模块并能单独测试**，不接入主流程。
**输入**：`backend/evidence/entries/*.json`（606+ 条，格式见第四节）。
**要做**：
1. 新建 `backend/evidence_store.py`：加载所有 entries → 用 `bge-small-zh`（或同级本地中文 embedding）向量化 → 建索引（可用 numpy 存向量 + 简单相似度检索，或轻量 faiss，不引入重型数据库）。向量结果缓存到本地文件，避免每次重算。
2. 提供函数 `search(query: str, topic: str = "", top_k: int = 5) -> list[dict]`：
   - 支持先按 `topics` 标签粗过滤再语义检索（标签为空则全库检索）；
   - 返回命中的证据条目 + 相似度分数，按分数降序；
   - **相似度低于阈值的一律不返回**（阈值先设一个可调常量，如 0.35）——这是"未命中降级"的判断依据。
3. 写一个 `if __name__ == "__main__"` 的自测：对几个测试 query（如"孕妇能吃蛋黄吗"、"空腹运动会掉肌肉吗"、"今天天气怎么样"）打印检索结果，验证：相关的能召回、无关的返回空。
**约束**：
- 纯本地、离线，不调用任何外部 API（embedding 用本地模型）。
- 依赖装到 `requirements.txt`，新增依赖要说明。
- 首次加载模型慢没关系，但要做成单例/缓存，别每次检索都重载。
**验收标准**（负责人会这样验）：
- 跑自测，"孕妇蛋黄"能召回膳食指南里蛋类/胆固醇相关条目；"天气"返回空（触发降级）；
- 检索单次耗时在百毫秒级；
- 代码是独立模块，没动 main.py 现有逻辑。

### 任务 2：单视频分析管线（接入检索层）

> 任务1验收通过后再做，届时负责人会给详细提示词。概要：新增单视频分析接口——
> 转写+画面 → 快模型拆「主张清单」→ 选定主张 → 调任务1检索证据 → 推理模型出结构化核验
> （判定/依据强度/更准说法/引用id 全由模型输出）。同时修 followup 的 max_tokens→8192。
>
> **接检索层的注意点（任务2实现时看）**：
> - `search()` 若传了 topic 但库里没有匹配该标签的条目会返回空 → 管线里应做兜底：topic 过滤为空结果时，退回全库检索（`topic=""`）再判阈值，避免标签不精确时把有效证据也压没。
> - 阈值 0.45 是"命中/未命中降级"的分界；search 返回空即触发"AI 常识判断，未收录依据"降级档。
> - 数据覆盖会随组员加库变好（如"鱼油"目前只召回到鱼肉相关，补充膳食补剂类文档后改善）——这是数据问题不是检索问题，管线逻辑不用为此改。

## 六、视频获取层决策（阶段3 实现，先记着别忘）

用户真实场景是"刷抖音拿到 URL"，所以 **URL 为主、文件上传仅兜底**。关键认知与决策：
1. **不下整段视频**：只下音频（几百KB，秒级）用于转写 + ffmpeg HTTP Range 抽几帧用于 OCR（现 main.py 已如此）。"下载慢"是伪命题，真正慢的是转写。
2. **最大提速**：云 ASR 替 Whisper（分钟级→秒级，阶段2）+ **演示服务器放国内**（抖音 CDN 在国内，下载音频快；放香港反而慢）+ 音频下载‖抽帧‖转写并行。
3. **抓取源双备份可切换**：TikHub（付费，现用）与 yt-dlp（开源免费，`-f bestaudio` 只下音频）都接上，哪个稳用哪个，一个挂了切另一个。yt-dlp 优先试。
4. **分析层与获取层解耦**：分析管线只认"本地音视频文件"，前面接可插拔获取层（文件上传/TikHub/yt-dlp）。获取方式坏了不影响核心分析。
5. 缓存/飞轮：同一视频链接二次进来直接返回上次核验结果（0秒），热门话题演示秒开（阶段5）。
